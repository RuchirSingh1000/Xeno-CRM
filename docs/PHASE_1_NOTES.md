# Phase 1 — Data model + realistic seed data

**Goal:** lay down the full database schema, set up migrations, and generate the messy multi-source seed data that Phase 2's identity resolution will operate on.

## What exists now

- **Full SQLAlchemy schema** (`app/models/core.py`) — 13 tables covering ingestion, canonical customers + identities, consent, orders, segments, campaigns, communications, communication events (append-only), webhook deliveries (idempotency audit), and AI runs (LLM call audit).
- **Alembic** wired to the runtime `DATABASE_URL`. Initial migration autogenerated. `render_as_batch=True` for SQLite (allows ALTER TABLE on dev).
- **Seed generator** (`app/seed/generate_csvs.py`) — produces four CSVs for the demo brand "Brewhouse Co.":
  - `brewhouse_pos_export.csv` (1,002 rows) — POS, phone-first, +91 format variance
  - `brewhouse_shopify_export.csv` (988 rows) — ecommerce, email-first, ~8% typo rate
  - `brewhouse_loyalty_export.csv` (973 rows) — loyalty, 10-digit phone + email + tier + DOB
  - `brewhouse_orders.csv` (5,000 rows) — cross-source order history
- **Underlying truth:** 1,500 unique people. ~30% in one source, ~40% in two, ~30% in all three. Identity resolution in Phase 2 will need to collapse ~2,963 rows back into ~1,500 canonical customers.
- **Data Sources page** (`/data-sources`) — brand panel, three source cards with previewable tables, downloadable CSVs, orders panel, and a note explaining why the data is shaped this way.
- **Sidebar nav** — phase-aware: Phase 0/1 pages active, future phases visible but disabled with a phase tag (signals roadmap to the reviewer).

## Demo defense notes

- **Why 1,500 underlying customers, not 100 or 100,000?** Big enough that identity resolution does meaningful work (~1,463 cross-source duplicates), small enough that the demo loads instantly and the marketer can scan individual customers. Trade-off articulated in TRADEOFFS.md (Phase 7).
- **Why deliberate dirty data?** Recruiters expect average submissions to seed clean dummy data. The bar for "exceptional" is data that *requires* the FDE skills the assignment is testing — schema mapping under inconsistency, identity resolution under conflict, data quality reporting that surfaces real issues.
- **Why this specific noise pattern?** Name spelling drift, phone format variance, and email typos are the three highest-frequency real-world data quality problems in Indian D2C onboarding. We don't simulate every possible problem — we simulate the ones that matter for the FDE story.
- **Why a separate `customer_identities` table?** Provenance. When a marketer asks "why did you merge these two records?" we need to show source system + raw values + normalized values + match confidence + reasoning. A flat denormalized `customers` table would lose this.
- **Why `communication_events` append-only?** Source of truth for state. Current status is derived (recomputed on event ingest). Lets the system handle out-of-order webhooks safely — Phase 4 will demonstrate this with explicit replay.
- **Why `ai_runs` as a table?** Every LLM call gets a row. Prompt version, raw output, validation status. Lets us answer "where did this AI decision come from" — a non-negotiable requirement for production AI in regulated markets like India.

## What's deliberately NOT in Phase 1

- No actual customer rows in the database. Phase 2 ingests the CSVs into `staged_records`, then identity resolution promotes them into `customers` + `customer_identities`.
- No consent rows yet — populated during ingestion in Phase 2 (we encoded the consent flags in the master pool though, so they're deterministic).
- No deployment. Local-first. Phase 7 will deploy.
